# 굳이굳이 웹에서, 일회성 컴퓨터를 정교하게 할당하는 웹 서비스를 만드는 법과 그 예시 (Application that Include PCOS Server의 응용법)

굳이굳이 웹에서, 일회성 컴퓨터를 정교하게 할당하는 웹 서비스이기에, 다음 기획을 한다.

## 기반이론 : PCOS

kernel : 기계어의 system call기능을 수행하는 라이브러리. system call이 호출하는 바.
프로세스 관리자 : 핵심 프로세스 (시작시 프로세스 관리용 프로세스 ; 즉 프로세싱 자원 관리라는), OS는 부트로더와 커널이 프로세스가 아니라, 그 다음 프로세스 관리자가 프로세스라 할 수 있다. 커널이하 LowLevel System을 Machine Code Level System라 하자면, Machine Code Level System위에 있는 그냥 소프트웨어는 Machine Code Level System를 호출 가능하다. 커널이 Machine Code Level System이다.
기본 초기 프로그램 : 프로세스 관리자 + Machine Code Level System를 Essence Of OS라 하자면, Essence Of OS는 기계제어+프로세스 관리를 하니, 실행 가능한 관리프로세스+기계제어 라이브러리이니, 최소한의 실행용 설비는 켜진거다. 기본 초기 프로그램이란, 관리자 프로세스가 맨 처음 키는 프로그램이다. 근데 사실 어플리케이션을 I/O로 받아서 실행하는 기능이 프로세스 관리자에 있다면 이건 필요가 없다. 에초에 프로세스 관리자는 프로세스를 관리하는 목적이지, 능동적으로 실행하지 않는다. 그러므로, 초기 프로그램은, Essence Of OS에게 프로세스 자원 및 컴퓨팅자원을 요청하는 "Essence Of OS이외의 소프트웨어"로, 시스템이 아닌, 시스템에서 첫 시작프로그램으로 지칭한 일게 어플이다.

system call : 컴퓨터의 단순 계산이 아닌 운영을 담당함
 - 외부 시스템 제어 (I/O(보조 저장장치 포함) 제어 기반)
 - 내부 시스템 제어 (사실상 RAM의 관리자는 시스템(OS + OS보다 LowLevel)임, 그리고 CPU에 프로세스를 할당하는것도 시스템(OS + OS보다 LowLevel)임)
file system : 보조 저장장치의 접근 포멧임

쉘이 시스템 아니냐 반박 : 시스템이 구동되기 위해 시스템에 내장된 어플이므로, 시스템이 아닌 어플리케이션SW이자 동시에, 시스템 자체의 작동상 필수적으로 요구되는 SW로, 컴퓨터에서 HW-SW는 굳이 한 SW를 올리면 그 HW는 해당 SW를 가동시켜야만 코드가 올바라 작동중이라 수 있으므로, 장치의 코드부분, 즉, 장치는 HW와 그 HW가 실행시키는 SW로 나뉘고, 그 중에서도, HW가 필수적으로 의존하는 SW가 코드부로, 시스템이 아닌 코드부다. 코드부의 정의는 그러므로, 장치의 주 프로그램이다. 코드부가 아닌 프로그램은 보조 프로그램인거다. 그러므로, 에초에 시스템-어플리케이션 분류가 웃긴 이유가, 에초에 실제로 컴퓨터 부팅후 실행된 코드는 주 프로그램인데, 그게 시스템과 쉘을 둘다 해당하기에, 나머지 보조 프로그램이, 그 하드웨어용 소프트웨어(코드부)로 지정된 상태가 아니라 해야한다. 그런데, 시스템은, 컴퓨터의 제어를 맏는 코드부, 쉘은 컴퓨터의 제어를 맏지 않는 코드부다. 즉, 에초에 컴퓨터 제어를 맏지 않는 코드부가 있는 순간, 그 컴퓨터는 걍, 쉘을 실행하는 실행기로 전락한거다. 그럴바엔, 시스템만 남기고 I/O로 코딩하는게 나은것 같다.
 - 쉘을 써야한다는건, 쉘 기반 범용 OS에 잠식된 사고인거다. 칩을 넣어서 개임하는 개임기면, 굳이 필요없을지도?

Q. 아예 기본 프로그램을 만드는 이유가 뭐냐? Machine Code Level System가 기계 제어 라이브러리면, 프로그램이 하나만 있어도 되지 않느냐?
A. 그래서 PCOS논의를 시작하겠습니다.

Machine Code Level System + 프로세스 관리자 = Essence Of OS잖아? 근데 그러면, 프로세스 관리자의 구현 코드를 걍, Machine Code Level System에서 커널용 syscall로 만들고, "Machine Code Level System + 커널의 프로세스 관리용 syscall의 권한을 유일하게 가진 프로세스 (프로세스 관리자)"꼴로 남기면, OS자체의 프로세스 제어를 유저가 커스텀 가능하니, (Partially) Customizible OS (PCOS)라 할수 있는거 아냐?

기계의 기능을 단일 기능으로 제어하는 앱(기계기능제한어플), 그러니까 기계에 앱 하나만 대응하고싶을때에, PCOS에서 프로세스 관리자가 앱 역할을 맡으면, 기계의 기능을 단일 기능으로 제어하는 앱(기계기능제한어플)을 만드면서도, 프로세스 자체의 한계를 주지 않고, 멀티프로세스 작업도 할수 있네? (Application that Include PCOS)

## 본론 : Application that Include PCOS Server

Q. 서버 자원을 유저가 임시로 일회성 할당받아 관리하는 서비스를 만들려 하는데, 어떨까요?
A. 그냥은 느려요, Application that Include PCOS Server를 논하겠습니다.

i. RAM은 빠르고 디스크는 느리다.

어짜피 삭제할 가상 디스크를, 고속 처리해야하며, 서버에서 클라이언트가 그 가상 디스크에 대해 조작해달라 서버에게 요청한다면, 가상 디스크가 RAM위에 있는게 나을듯.

ii. 커스텀 철학 : 모든 소프트웨어는 유저가 커스텀 가능해야 좋은 소프트웨어다 ; 일회용이지만, 유저가 저장을 원한다면?

파일 시스템 포멧 A(Dump Of Ephemeral Filesystem)와, 어짜피 RAM에서 구동중이라 A에서 일부 저장정보를 빼도 되는 상태 B(Ephemeral Filesystem, 해당 프로세스에서 자명한 정보를 생략했기에, 이걸 그대로 반환하는 순간 다음 프로세스가 이걸 처리 못함 주의)에 대해, 유저가 해당 가상 OS의 저장을 원한다면 B에서 A로 덤프해서 "서버상 유저의 StaaS"에 저장하면 그만이잖아.
 - 유저 다운로드는 시간이 너무 많이 걸림
 - 이건 UX가 확실히 평범함.

1. 가장 확실함.
2. 실시간 저장이나 이런 교묘한 기능...? 그런 테크닉은 요구될때 만드는게 나으며 (산업은 일단 작동을 원하는 더러운 새끼들), 에초에 저장 기능 제공에 감사해야지 왜 느리다 생각함?
3. 긴급 저장 기능은 메인이 아니고, 비상용이라, 빠르기보다 확실성이 중요함.

iii. RAM과 디스크가 정해졌다면 구체적 구조가 나와야겠지?

 - 가장 어려운 장벽 : 컴퓨팅 자원 할당시 창조가 아닌 할당이고, 심지어는, 할당한 자원 관리마저도, 엄청난 고생임.

 - 그것만 "대충 때우면", 일단 작동하는 프로그램임.

1. StaaS : 디스크 용도. 원래 휘발성 OS라 필요 없지만, 유저가 StaaS연동기능을 활성화하면, 우리가 StaaS에게 짬처리, "긴급 저장 및 종료"라는 행동을 통해서, Dump Of Ephemeral Filesystem을 여기에 저장하고, "긴급 저장 노트북 오픈"라는 행동을 통해서, 이전에 저장한걸 불러옴.
2. RAM의 Ephemeral Filesystem : PCOS가 접근할 보조저장장치.
3. RAM의 Ephemeral Filesystem이외 : 그냥 RAM임.
4. Application that Include PCOS : 할당된 RAM가지고 어플 역할을 함. 입출력은 I/O에서 받음.
5. Application that Include PCOS Server : 유저가 직접 IaaS서비스 자원을 주면, i. 받아먹고 (이 서버 자체는 IaaS가 아니므로), ii. 그 서버에다가, 유저 요청이 "새 노트북 오픈"이나 "긴급 저장 노트북 오픈"이면 Application that Include PCOS를 열어주고, iii. 그다음, 서버 자체는 Application that Include PCOS에 접속하여, 유저가 Application that Include PCOS를 조작하는걸 반영하거나, 노트북에다가 "저장 및 종료"나 "영구 완료"를 누르면, 그 서버에가다 덤프 요청을 보낼지 말지 앞선 입력으로 판단하여, StaaS에 저장 시도를 하고, 성공 여부를 다시 Application that Include PCOS에 보내서, 실폐시 다시 시도하도록 유저에게 권유하도록 하고, 그 다음 해당 Application that Include PCOS를 영구 완료하는 방식 ~~북한 애들이 완료라고 하는거 우스광스럽고 폭력적이라 채택함 ㅋㅋㅋㅋ 북한 그저 웃음밸.~~
 - 모듈화/분업화 : PCOS할당 관리를 서버가 대신 해주면 된다는 논리
 - 비용전가 : 유저가 하드웨어를 직접 넣어야 함.
 - 비용 최소화 : 우리는 저전력 서버로 만들 생각임. 그 태크닉은 어케 하냐고? 프로그래머가 존나 똑똑하면 됨. 산업적으로 무리 맞음. 그러나 기술적으로 하드케리하면 됨. 시간이 존나 낭비되겠지.
  - 커스텀 기능 : 유저는, 새 노트북을 만들때, 어플 겸 프로세스 관리자를 뭘 쓸찌를, 인터넷 파일로 링크하거나, 에초에 전용 오픈소스 저장 플랫폼을 제공할거라 그 저장소를 사용.
  - 우리의 역할 : 대충 자원 넣으면 어플 나오는 플랫폼.

즉, 굳이굳이 웹에서, 일회성 컴퓨터를 정교하게 할당하는 행위가, 웹의 시대가 열리며, 원거리에서 컴퓨팅이 중요해짐에 따라, 수요가 늘 가능성이 있음.

이 모든 서비스는 오픈소스로 진행하고, 서버 자체의 운영은, 유저들이 보는 광고 비용으로 수익을 얻을까 함.

그리고 난 에초에, 굳이굳이 웹에서, 일회성 컴퓨터를 정교하게 할당하려는 어플 개발 목적이 있거든.

## 실재응용 : 이성 시뮬레이터

문서란, 문단들의 연속인, 한 "page"를 최소단위로 가지는것이 보통이다. 그렇기에, markdocs가 저장된 컴퓨터에서, 특정 맥락상 단어를 지칭하는 주소를 다음과 같이 구상 가능하다.

1. 파일경로
2. markdocs파스트리상 page의 위치 (xpath)
3. 그 page내에서 문단에 대한 지칭 (보편명제긍정론적 인덱스)

3번은 Paragraph에 대한 접근으로, 원래는, 보편명제긍정에서 본인은 ℙᵢⱼₖ라 적는다.

1. i는 문단 주소로, 글의 형식이, 문단 전 한깐 띄어쓰기를 감지하는거다.
2. j는 문장 주소로, 온점을 감지하는거다.
3. k는 문장 내 조작으로, 쉼표를 기준으로 인지한다.

글을 읽을때, "문단화"/"온점화"/"쉼표화"라는 과정을 거쳐야 한다. 이 부분은, 거친 다음에, 그러한 의도가 맏는지 판단하는 방법은

1. 글쓴이가 정상적으로 썼다면, 이성을 통해 파악할수 있다. (이성을 써야하는데 이성이 필요하다.)
2. 글쓴이가 정상적으로 썼다는 가정은, 폭력적인 전재이다. 그렇지 않고 유저에게 확인하는것이 옳다.

그 다음, "언어의 사용은 의미에 있다."

Pragmatics Approximation Model Interface라는 유닛에, 외부 소통용 동적 라이브러리를 단다.

그러면, Pragmatics Approximation Model에 접근하는 체널을 통해, 의미를 얻을수 있다.

Pragmatics Approximation Model는 두가지 종류로 나뉜다.
1. 완전 반영 : Pragmatics Approximation Model를 호출하면 인간상담사가, 질문에 응답한다.
2. 물량 근사 : Pragmatics Approximation Model를 호출하면 LLM상담사가, 질문에 응답한다.

LLM은 언어모댈이 아니다, 화용론을 근사한 덩어리일 뿐이다. 그걸 숭배하는건 역겹기 그지없다.

나는 AΙ혐오자다. AI에게 이성적 사고영역은 존재하지 않는다.

그런데 나도 인간이지만 이성적 사고영역이 극단적으로 저하되어서, 사람들의 행동을 암기한다. 초 4 때 기억도 있다. 그래서...

내가 만든 이성 시뮬레이션을 실현시키는거다.

암튼... 이성 시뮬레이터의 구조는 크게 세가지와 두 부분으로 나뉜다.

i. 기초 레이어
1. Pragmatics Approximation Model Layer
2. DLPFC Emulating Lanauge
3. Simmulating Agent
ii. 기억 레이어
1. 휴리스틱(Heuristic)이라는 인간의 기능을 시뮬레이션하는, 전용신경망

DLPFC Emulating Lanauge가 하는일은 뭘까?

i. 맥락파악
1. markdocs문서에서, 현재 문장과 문단을 본다.
2. 어떤 주제와 이후 내용을 함의하는가?라는것에 대하여, 가설을 주입받으면, 그거에 대해, "이성 시뮬레이터"를 호출한다.
3. 문단이 여러개를 봤는가? 그렇다면, 그것이 함의하는 의미를 찾는다. 이걸 하기 위해, "이성 시뮬레이터"를 호출한다.
4. 이런 과정을 통해 사실적 읽기만 완료된다.
ii. 의미분석
1. 문장을 보고, "해당 맥락상, ℙᵢⱼₖ의 뜻은 ℙₗₘₙ이다"가 정확히 나오도록, "정형 markdocs화 글의 표면적 속뜻(맥락적 속뜻이 아닌것)의 의미파악용 이성추론용 markdocs"를 돌린다.
2. 그렇기에, 사회 통념상 yes/no나 속뜻 해석이 명확한 경우, Pragmatics Approximation Model Layer가 그 뜻을 확정한다.
3. 명확해질때까지 무한시간을 소모해서라도 이성을 돌린다. ("이성은 진리를 재생산한다"... 보편적 설득력은, 보편적인 문장이거나, 보편적인 문장에서 개연적(맥락정합적)으로 유도된 문장에서 나온다. 즉, 이는 진리의 재생산이다.)

#### 이 기술이 의의

1. 시간이 너무 느린 정도 : 프로그램 < 기존 AI <<< 이성 에뮬레이터 (무한시간이 될수 있고, 그게 무한인지 아닌지도 정지문제)
2. 정확도 : 프로그램 = 0 < 기존 AI (40%도 안되고 10 이하라 봐야겠지. 수치적인 부분을 뺴면 죄다 흉내인데 0을 안준게 어디냐) <<< 자동화된 이성 에뮬레이터 (에이전트와 화용론 근사 모델이 모두 LLM) ≈ 인간이 직접 따라한 이성 에뮬레이터 = 100%
3. 정확도 세부 : 자동화된 이성 에뮬레이터 (에이전트와 화용론 근사 모델이 모두 LLM) < 인간이 직접 따라한 이성 에뮬레이터 = 100%

이를 통해 알수있는 사실
1. 시간을 배제하고 가장 정밀하고 이론상 완벽한 모델은? : 이성 에뮬레이터이다.
2. 가장 효율적인 기술은? : 기존 AI다.
3. 우리가 만들 기술의 목표는?

그 목표는 바로, "앙상블을 쓰든 뭐든, 수단과 방법을 가리지 않고, 속도가 빠르며, 속도가 이성 에뮬레이터마냥 좆되지 않되, 성능이 이성 시뮬레이털르 따라잡는것" ⋯ (이상적 인공 이성)

즉, 이상적 인공 이성의 구현이다.

### 이론적 레퍼런스 : [내 언어철학관](https://lrl.kr/dGwOz)

````markdown
# 보편명제 증명론 (N.B. 모두 가설에 영역이다. 이 증명론이 실제로 작동하는지는 모른다.)

인과관계란 "SI P, ERGO Q"가 가지는 관계를 말한다.

N.B. 자연어에서는 "EXSTAT ALIQUIS P, Q, SI P, ERGO Q. NON HOC EST, NON L VEL Q"다. 다른 경우가 존재한다. 칸트의 정언명령과 그 공리는, 자연어가 사용하는 "맥락(CONTEXTOUS)"에 따른 인과관계를 써야한다.

함축관계 : 무맥락 인과관계
무맥락 인과관계 : "NON P VEL Q"를 이르는 말

## 절대적인 규칙 : 논리의 정의에 따라서 보편적으로 참인 문장

"SI QUOD ILLUD, SI P, ERGO Q, EST VERUM ATQUE IN ID CONTEXTOUS, SI NON P VEL Q, EST NON REPUGNAT, ERGO, NON P VEL Q"

즉, 기존 문맥과 맥락 정합적일때만, 전건긍정이 성립한다.

## 문장의 해석

자연어를 해석하는 구문을 자연어로 작성하면 "Q. p의 뜻은 q인가? A. yes"이다.

자연어에서는, 화자와 청자가 있으니, 스스로에게 질문한다면, 긍정 여부를 알수 있다.

## 증명 이론 : 문장이 함축하는 의미의 파악

그걸 자연어로 쓰면 "p이다. Q. 그렇다면 q인가? A. yes"의 형태가 된다.

보편 명제의 증명은, "SI QUOD ILLUD, SI P, ERGO Q, EST VERUM ATQUE IN ID CONTEXTOUS, SI NON P VEL Q, EST NON REPUGNAT, ERGO, NON P VEL Q"로 진행되기 때문이다.

## 문맥 스택 (Contextous Stack)

문맥 의존 언어는, 최소한 push-down automata로 작동한다.
그러므로, "ATQUE IN ID CONTEXTOUS, SI NON P VEL Q, EST NON REPUGNAT"라는것은, 이전 문맥을 참조한다.

그러므로, 문장은 Contextous Stack에 보편명제 증명론에 의해 「사고를 이치에 맞게 이끌어 가는 과정 (Step by Step)」을 타당하게 적었다 볼 수 있다.

## 형식화된 보편명제의 증명

사고에 있어서, `1. 2. 3.`이라는 스탭별로 진행한다.

문단의 리스트 Paragraph에 대해, Paragraph[i]는 i문단이고, Paragraph[i][j]가 i문단의 j문장일때,

ContextousStack[StackPointer] = "Paragraph[i][j]의 값은 p다."
ContextousStack[StackPointer + 1] = "Q. Step1 Question"
ContextousStack[StackPointer + 2] = "A. Step1 Answer, Q. Step2"
...
ContextousStack[StackPointer + n] = "UT, Step_n"
식으로 추론된다.

이는 배외측전전두피질(DLPFC)가 인간에 뇌에서, Step별로 추론을 하기 때문이다.

굳이 Step시스템을 쓸 필연성은 없다. 물론 필요성은 있을수 있다.
````

시한부 레이어는, 정확 추론을 여러번 해야 생기고, 정확 추론은 LLM이 안망가지고, 휴리스틱도 망가지지 않고 (에초에 휴리스틱같은 경우 시한부 레이어와 달리 최소한의 준정밀한 검증이 있으며, 오염을 막는 설비가, 에초에 이게 오염될 이유도 없을정도로, 휴리스틱이 추론에다가 완전히 반영되지 않고, 그냥 인간 수준의 오해말고는 오염될 일도 없는데, 그런 오염방어구조가 존재.), 그래서, 시한부 레이어는, 숙달이 됬다면, 그때부터 주입식교육에 대한 대답을 하는 셈이라, 그따구인것.
